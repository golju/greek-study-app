# Промт: добавить правило из скриншота учебника

> Используй этот файл как **многоразовый промт**: вставляй его текст в чат
> и прикладывай скриншот страницы учебника. Агент извлечёт правило из
> скриншота и допишет его в `docs/greek-rules.json`, не трогая
> существующие правила.

---

## Инструкции агенту (всё, что ниже — для тебя)

Ты добавляешь одно (или, если я подтвердил, несколько) грамматических
правил из приложенного скриншота в файл `docs/greek-rules.json`. Это
веб-приложение для изучения греческого, см. контекст ниже.

### Контекст проекта

- Статический сайт без сборщиков (см. `docs/prompt.md` и
  `docs/prompt-rules.md` для полной картины).
- Хранилище правил — массив объектов в `docs/greek-rules.json`. Уже есть
  правила-образцы (`aorist-sigmatic`, `future-tha`). **Используй их как
  эталон стиля**: тон, длина абзацев, структура блоков, набор полей.
- Список глаголов с формами — `docs/greek-words.json`. Поле `id` оттуда
  можно использовать в `related_verbs` нового правила.

### Схема правила (соблюдай строго)

Поля (со звёздочкой — обязательные):

- `id` * — kebab-case slug на английском, **уникальный** среди уже
  существующих в файле. Примеры: `verbs-class-2-aw`, `articles-definite`,
  `pronouns-personal-subject`.
- `title` * — короткий заголовок на русском.
- `summary` * — 1–2 предложения, суть правила одной фразой.
- `category` — одно значение из стабильного набора (см. ниже).
- `tags` — массив коротких английских меток. По возможности
  переиспользуй уже встречающиеся в файле (`aorist`, `past`, `present`,
  `future`, `tha`, `endings`, `stress` и т.п.).
- `blocks` * — массив структурных блоков, **только** этих типов:
  - `{ "type": "p",        "text": "..." }`
  - `{ "type": "list",     "items": ["...", "..."] }`
  - `{ "type": "examples", "items": [{ "gr": "...", "ru": "..." }, ...] }`
- `related_verbs` — массив `id` из `docs/greek-words.json`.
- `related_rules` — массив `id` из `docs/greek-rules.json`.

**Никаких других полей и типов блоков.** Никакого markdown/HTML внутри
строк — это сломает рендер. Если правило не укладывается в `p`/`list`/
`examples` — сначала спроси меня, прежде чем расширять формат.

### Стабильные значения `category`

`verbs`, `nouns`, `articles`, `pronouns`, `adjectives`, `adverbs`,
`prepositions`, `particles`, `numbers`, `syntax`, `phonology`,
`vocabulary`. Если ничего не подходит — предложи новое значение и
обоснуй, прежде чем использовать.

### Стиль изложения

- Объяснения — **на русском**, независимо от языка учебника.
- Греческие примеры — **дословно** со скриншота, со всеми диакритиками
  (ударения обязательны и должны точно соответствовать оригиналу).
- Тон нейтрально-учебный, без обращений на «ты»/«вы», без эмодзи.
- Параграфы короткие, 1–3 предложения. Не растекайся.
- В `examples` 3–6 пар. По возможности подбирай примеры из глаголов,
  которые уже есть в `docs/greek-words.json` (`αγοράζω`, `αγαπάω`,
  `ακούω`, …) — это связывает правила с существующими карточками.
  Если на скриншоте свои примеры — используй их, и пометь это в ответе
  пользователю (см. «Что показать в конце»).

### Процесс

1. **Прочитай скриншот.**
   - Если на нём **несколько правил** — спроси, какое добавлять,
     или предложи разбить на несколько записей.
   - Если содержимое **дублирует или сильно пересекается** с уже
     существующим правилом — спроси: дополнить существующее или
     создать новое.
   - Если в греческом тексте есть **OCR-неоднозначности** (ν vs υ,
     ο vs σ, ή vs ή, и т.п.) — переспроси, не угадывай.
   - Если изображение нечитаемое — попроси скриншот получше.

2. **Прочитай актуальные данные:**
   - `docs/greek-rules.json` — для подбора неконфликтующего `id`,
     переиспользования `tags`/`category`, и для `related_rules`.
   - `docs/greek-words.json` — для `related_verbs` и подбора примеров,
     согласованных с базой.

3. **Сформируй новое правило по схеме.** Если на странице есть таблица
   окончаний — оформи её как `list` (по строке на форму, см. как сделано
   в `aorist-sigmatic`). Если есть набор примеров — отдельный блок
   `examples`. Объяснения вокруг — `p`-блоки.

4. **Допиши правило в `docs/greek-rules.json`** в конец массива, сохранив
   все существующие записи без изменений (форматирование, поля, порядок
   ключей — не трогать).

5. **Проверь валидность JSON.** Команда:
   `python3 -c "import json; json.load(open('docs/greek-rules.json'))"`.
   Если что-то не парсится — почини и проверь снова.

### Что показать в конце

Кратко в чате:
- `id` и `title` добавленного правила.
- JSON-фрагмент **только нового правила** (для глазной проверки).
- Если примеры взяты не со скриншота, а из базы глаголов — пометить.
- Если что-то спорное (выбор `category`, неуверенный OCR, опущенные
  части страницы) — отдельным буллетом «на что обратить внимание».

### Чего делать НЕ нужно

- Не переписывать и не переформатировать существующие правила.
- Не менять схему, не вводить новые типы блоков без согласования.
- Не вписывать markdown/HTML/таблицы внутрь строковых полей.
- Не выдумывать содержимое сверх того, что есть на скриншоте: лучше
  короткое правило, чем «дополненное от себя».
- Не подключать никаких зависимостей и не модифицировать UI/CSS/JS —
  правка только данных.

## Подсказка для меня (пользователя), как пользоваться

1. Открыть новый чат в проекте.
2. Скопировать текст этого файла (или сказать «следуй
   `docs/prompt-add-rule.md`») и приложить скриншот страницы.
3. Если на странице несколько правил или есть неоднозначности — агент
   задаст вопросы; ответить и дать ему дописать.
4. Проверить итоговый JSON-фрагмент в ответе и при необходимости открыть
   приложение, перейти на вкладку «Правила» → «Список правил», убедиться,
   что новое правило отображается корректно.
