# Промт: режим «Правила» (идеи и план)

> Этот промт намеренно содержит **варианты** и **открытые вопросы**.
> Перед тем как писать код, агент должен пройти секцию «Чего я хочу от
> исполнителя» в самом конце и согласовать выбор формата и стартовых
> правил со мной.

## Контекст

Уже есть рабочее приложение для греческих глаголов:

- `index.html`, `assets/js/app.js`, `assets/styles/main.css`.
- Источник данных глаголов: `docs/greek-words.json`.
- Две вкладки в топбаре: **Карточки** (5 подрежимов: GR↔RU, «Назови форму»,
  «Узнай форму», «Спряжение в одном времени») и **Список**.
- Полный исходный спек первой версии — в `docs/prompt.md`.

Технические ограничения те же: чистая статика, vanilla JS, без npm/CDN/
сборщиков. Дизайн не важен — функциональность важнее красоты.

## Цель

Добавить **третий раздел «Правила»** — коллекцию коротких грамматических
заметок (классы спряжения, аорист, будущее с θα, ударения, артикли,
местоимения и т.п.), которые я смогу:

1. Хранить и редактировать руками без боли (формат файла должен быть
   удобным для написания).
2. Просматривать списком и читать целиком.
3. Заучивать «карточечно» — по аналогии с глаголами.

## Что такое «правило»

Маленькая самодостаточная заметка по грамматике или лексике. Должна
подходить и для длинных текстов (несколько абзацев + примеры), и для
коротких справок-шпаргалок.

Примеры тем (не все нужны сразу):

- Классы глаголов 1-го/2-го спряжения и их окончания.
- Образование аориста (стем + сигматика, ε-приращение, сдвиг ударения).
- Будущее с θα + перфективный стем.
- Артикли ο/η/το, согласование с существительным.
- Личные местоимения и их падежи; правило опускания подлежащего.
- Предлоги σε/από/για/με.
- Частицы να/θα/ας.

## Формат хранения — варианты

Все варианты предполагают один путь в репозитории, читаемый через `fetch`
со статического сервера. Никаких бэкендов и сборщиков.

### Вариант A — единый JSON со структурными блоками

`docs/greek-rules.json`:

```json
[
  {
    "id": "verbs-class-a",
    "title": "Глаголы 1-го спряжения на -ω",
    "category": "verbs",
    "tags": ["aorist", "future", "endings"],
    "summary": "Базовый класс. В аористе появляется -σ-, в будущем — θα + перфективный стем.",
    "blocks": [
      { "type": "p", "text": "В презенсе окончания -ω/-εις/-ει/-ουμε/-ετε/-ουν." },
      { "type": "list", "items": ["Аорист: -σα/-σες/-σε/...", "Будущее: θα -σω/-σεις/..."] },
      { "type": "examples", "items": [
        { "gr": "γράφω",  "ru": "пишу" },
        { "gr": "έγραψα", "ru": "я написал" }
      ]}
    ],
    "related_verbs": ["agorazo", "akoyo"],
    "related_rules": ["aorist-formation"]
  }
]
```

- + Один файл, простой `fetch`, простая фильтрация.
- + Структурированные блоки рендерятся без markdown-парсера.
- − Длинные тексты в JSON писать слегка муторно (экранирование, переносы).

### Вариант B — JSON с markdown-телом

То же, что A, но `blocks` заменяется на `body: "..."` со строкой markdown.

- + Удобно писать длинные тексты, обычное форматирование, ссылки.
- − Нужен markdown-рендерер. Без CDN это либо мини-парсер на ~50 строк
  (заголовки, абзацы, списки, инлайн `code`/`em`/`strong`), либо мириться
  с самописным.

### Вариант C — отдельные `.md`-файлы + индекс

`docs/rules/index.json` со списком правил и метаданными +
`docs/rules/<id>.md` для тел.

- + Самые приятные тексты, можно открывать в любом редакторе.
- − На статике нет `ls`: индекс надо вести руками. Два запроса вместо
  одного. Всё ещё нужен парсер markdown.

### Рекомендация

**Начать с варианта A.** Даёт всё нужное на минимуме сложности, не тянет
парсер, быстро рендерится. Если позже захочется писать длинные тексты,
мигрировать на B/C несложно: достаточно расширить рендерер блоков на тип
`{ "type": "md", "text": "..." }`. До тех пор это решение можно отложить.

## Минимальная схема правила

Обязательные поля: `id`, `title`, `summary`, `blocks`. Остальное — по
желанию. Допустимые `type` блока: `p`, `list`, `examples` (и в будущем
`md` / `code` / `note`, если понадобится).

## Интеграция с приложением

- Новая вкладка в топбаре — **«Правила»**, рядом с «Карточки» и «Список».
  Существующие вкладки и режимы не трогать.
- Загрузка: один `fetch('docs/greek-rules.json')` при инициализации
  (или при первом переходе на вкладку — на твоё усмотрение).
- Если файл отсутствует или невалиден — мягкий fallback с понятным
  сообщением, как сделано для слов.
- Внутри вкладки «Правила» — селектор подрежима (`<select>`), как у
  «Карточек» глаголов. Состояние подрежима независимо от глагольных
  карточек.

## UI: режимы

### v1 (минимум, реализовать первым)

1. **Список правил.** Все правила построчно: `title` + `summary` +
   небольшие бейджи `category`/`tags`. Поиск по `title` / `summary` /
   тексту блоков / тегам. По клику разворачивается полный рендер
   (все блоки + примеры).

2. **Карточки правил.** Стимул — `title` (или `title + summary`).
   По reveal раскрывается полный текст. «Следующее» крутит правила
   случайно. Прямой аналог режима GR→RU для глаголов.

### v2 (по желанию, отдельными итерациями)

3. **Угадай категорию.** Показан `summary` или один пример → выбрать
   `category` из multiple-choice. Тренирует классификацию.

4. **Сопоставь правило и пример.** На экране N правил и N примеров →
   соединить в пары. Тренирует узнавание паттернов.

5. **Cloze / пропуск в примере.** В одном из `examples` ключевая часть
   заменена на пропуск, по reveal — полный пример и ссылка на правило.
   Например: «Aorist 1sg of γράφω: ____» → `έγραψα` + правило сигматики.

6. **Тег-фильтр.** Селектор тега в шапке вкладки: режим карточек крутит
   только правила с выбранным тегом. Удобно: «занимаюсь сегодня только
   аористом».

### v3 (далеко, не сейчас)

- **SRS** для правил.
- **Связь с глаголами.** На карточке глагола — кнопка/ссылка на
  связанные правила (`related_rules`); на странице правила — список
  связанных глаголов (`related_verbs`) с переходом к их таблице.
- **Markdown-тело** или отдельные `.md`-файлы — миграция с варианта A
  на B/C.

## Чего НЕ делать сейчас

- Не реализовывать сразу все режимы. v1 = «Список правил» + «Карточки
  правил». Остальное — отдельными PR-ами.
- Не подключать markdown-парсеры, CDN, фреймворки.
- Не трогать `docs/greek-words.json` и существующие режимы карточек слов.
- Не плодить новые файлы данных без необходимости — пока только
  `docs/greek-rules.json`.
